# FactorAgent2 执行过程反馈（blockers / 口径不清晰点 / 改进项）

[2026-01-23 05:11:48 UTC+8] 本文件用于记录本轮任务执行过程中曾经阻塞进度、耗时修正、以及口径不统一/不清晰导致的风险点，并给出我对后续工作方式的改进方案（避免“写出来 ≠ 预想”的问题再次发生）。

## 1) 曾经 block 住或消耗大量时间的问题

### 1.1 `trade_bs_flag` / `side` 字段不可靠（最核心 blocker）
- 现象：依赖 `trade_bs_flag/side` 做成交主动方方向判断时，会导致 buy/sell hit、wall 侵蚀归因、trade 相关不对称等信号方向性错误。
- 影响：属于“系统性口径错误”，一旦用于多个信号簇，会导致整套因子集方向性偏差，后续所有刷数/报告需要整体作废重跑。
- 修复：按最新指令改为使用 `bid_id` 与 `ask_id` 比较推断主动方（id 更大的一方为主动方），并重刷因子；原基于错误口径的旧数据与报告全部作废。

### 1.2 数据源/刷数流程的“环境一致性”问题
- 现象：若没有严格按 `setup_sdk.sh` + `playground run-agent` 的流程初始化环境/刷入 `factor_pool`，容易出现“能编译但刷不出/刷不齐”的情况（路径、参数、loader 不一致）。
- 风险：同一份代码在不同 shell/不同数据根目录下刷数口径可能漂移，导致结果不可复现。
- 处理：本轮明确固定为 **parquet 数据源**（`/data/share/dev/hft/data/market_data_parquet`），并把所有可复现命令收敛到交付工程 README。

### 1.3 与 `basic_table` 对齐（time/md_id）与“每 tick 输出”的工程细节
- 现象：若 `time/md_id` 与 `basic_table` 不对齐、或遇到空簿/crossed 时直接 `return`，会造成 tick 丢失/无法 join、以及下游分析样本不一致。
- 修复要点：
  - `md_id` 采用 `biz_index`（order/trans）以对齐 `basic_table`。
  - `time` 使用 `event.local_ts` 基础上对齐 `basic_table` 的现有口径（见强验证报告中的说明）。
  - **每条 MarketEvent 都落一行**：即使 book 为空也输出（填确定默认值），确保 schema 稳定且可对齐。

### 1.4 内存/并发设置需要反复试错确认（避免超 180GB）
- 现象：全 code 单进程回放 RSS 峰值可到 ~33GB（与日期/活跃度相关），并发一旦激进容易触顶。
- 处理：本轮月度刷数与全历史刷数均采用 `workers=4` 的保守并发，并在 `research_log.md` 里记录预算与观测，避免“先跑起来再说”。

## 2) 口径不统一/不清晰点（需要文档或框架侧进一步澄清）

### 2.1 文档默认口径与真实数据字段可靠性冲突
- `FactorAgent2` 任务文档中对 trade 方向的定义示例使用 `trade_bs_flag`，但本数据集该字段已确认存在问题。
- 建议：在 `factor_agent_docs` 或 `factor_docs_validation` 中显式标注“当 bs_flag 不可靠时的 fallback：`bid_id/ask_id` 判定主动方”，并注明优先级。

### 2.2 “对齐 basic_table”到底是“行数一致”还是“键一致”
- 在验收中，行数对齐是必要条件，但真正可用性来自 `(code,time,md_id)` 的键对齐（可 join）。
- 本轮做法：用行数对齐（含连续交易时段）作为硬验收；若后续发现存在键漂移，应补充“抽样键匹配率”验收。

## 3) 为什么此前 Step2 强验证会漏掉这么明显的方向错误？本轮如何改进

### 3.1 根因（同源复算导致“验证失效”）
- 之前的实现与点对点复算若同时依赖同一个错误字段（例如都用 `trade_bs_flag/side`），那么即使逐点对比也会“错得一致”，从而无法暴露问题。

### 3.2 本轮改进（把验证从“复用同源逻辑”改为“多路径互证”）
- 主实现改为 `bid_id/ask_id` 推断主动方（不再依赖有问题字段）。
- 强验证新增/强化：
  - 与 `basic_table` 的逐日行数对齐（尤其是连续交易时段）作为硬门槛，防止 tick 丢失。
  - 点对点复算使用独立复算 Agent；抽样覆盖 ≥200 code、≥10,000 点，并逐信号统计 pass_rate。
  - 增加非退化 sanity check（例如 hit_rate 最大值 >0、book_valid 最大值=1）。
- 目标：即使未来再出现“字段口径异常”，也能在强验证阶段被及时捕获并阻断后续全历史与报告产出。
