# Session Loose-Ends — 会话遗留自检

回答"本会话还有遗留吗?有没有更新到 issue 的内容、没提交的代码、没 merge 的 PR?"
把本会话**所有对外/半持久动作**逐项核实当前状态,分类列给用户判断。

> 定位:收尾自检,不是全仓体检。**只查本会话自己产生的东西**,不把仓库里别人的/历史的脏状态当遗留。
> 红线:**全程只读**,不 commit / 不 push / 不 merge / 不删 / 不 stash / 不改任何文件——只报告,让用户拍板。

## 使用方式

```
/session-loose-ends            # 审计本会话遗留并分类列出
```

## 为什么在主进程做(不派 subagent)

这个自检依赖**本会话的上下文**——只有主进程记得"这会话到底做了哪些动作"。
subagent 是新上下文,不知道本会话干了什么。故本 skill 在主进程执行:先从对话回忆动作,
再用**只读** git / gh 核实每项当前状态。(核实是只读的,不涉及起草长文本,无需外派。)

## 执行流程

### Step 1: 从对话回忆本会话的"对外/半持久动作"

不看代码、先回忆。列一张本会话干过的清单(逐项后面要去核实):
- **代码改动**:编辑/新建过哪些文件、在哪个仓。
- **git**:建过哪些分支、commit 过什么、push 过哪些分支。
- **PR**:开过 / 改过 / 合过哪些 PR。
- **issue**:建过 / 评论过 / 关过哪些 issue;**讨论中拍板过的决策、结论、实验结果**——有没有落到某个 issue 上。
- **发版 / 部署**:打过 tag / 发过 release 吗?发了有没有部署?说好要部署的做了没?
- **凭证 / 密钥 / 敏感产物**:生成过 key / token / secret 吗?临时私钥、含密码的文件落在哪(scratchpad?),清了没?
- **memory**:本会话冒出的**耐久且不易从仓库/issue 推得**的事实(架构决策、非显然的运维知识、用户反复纠正的偏好),该存 memory 的存了没。
- **临时文件**:往 scratchpad 写过什么,尤其**敏感内容**。

### Step 2: 逐项只读核实当前真实状态

公司仓(`fin-infra/*`、`hft-prop/*`)的 gh 操作必须带企业 token,gh 常在 `~/anaconda3/bin`:
`export PATH="$HOME/anaconda3/bin:$PATH"; export GH_TOKEN=$(gh auth token --user ken-chen_scale)`
(`gh issue view` 有时吞输出,改用 `gh api repos/<owner>/<repo>/issues/<n>` 更稳。)

对每类动作,确认它是否走到了预期的**终态**:
- **代码改动** → 已 commit?已 push?进了目标分支(常是 `origin/main`)吗?
  - `git log <branch> -- <file>` / `git show origin/main:<file> | grep ...` 确认改动确实在目标分支。
- **本地 commit** → 有没有 commit 了却没 push 的(`git log origin/<branch>..<branch>`)。
- **push 的分支** → 有没有 PR?(`gh api repos/.../pulls?head=<owner>:<branch>&state=all`)
- **PR** → merged?还是 open 未合?(`gh api .../pulls/<n> --jq '{state,merged,mergeable}'`)
- **合过的 PR / 关掉的 issue** → 有没有**该做的后续没做**:发版?部署?CHANGELOG?**相关 tracker/母 issue 要不要补一句"已解决"**?
- **决策/结论** → 是否已落到某 issue(open/update/conclusion)。只在脑子里/对话里 = 遗留。
- **tag / release** → 发了吗?**发了有没有部署**(若部署是既定下一步)。
- **密钥/敏感文件** → 私钥已 shred/删?scratchpad 无密钥残留(`ls scratchpad | grep -iE 'key|pem|token|deploy|id_'`)?
- **memory** → 待存事实是否已写(对照 `MEMORY.md` 索引)。
- **本地分支** → 本会话建的临时分支,合并后清了没(`git branch`)。

### Step 3: 并发安全甄别(重要,别误报也别乱动)

本机常**多会话并发操作同一仓**(切分支 / reset / 改文件)。核实时:
- 当前分支、HEAD 可能**已被别的会话切走/推进**——别假设还停在你离开时的位置。
- `git status` 里的未提交改动、`git branch` 里的其它分支,**未必是本会话的**。
  只有**能对应上本会话确实编辑过的文件**,才算本会话遗留;
  对不上的一律归为"**他人/环境状态,非本会话,未触碰**",明确点出但**绝不 commit / stash / 还原**。
- 拿不准某改动是不是自己的,按"不是自己的"处理并如实说明,请用户判断。

### Step 4: 分三类列出,然后停手

按下面三桶输出,每条给出**当前状态 + 定位链接/路径**;列完**停止,等用户逐项判断**,不自行动手:

- **✅ 已闭环**(供核对,证明不是遗留):merged 的 PR、已发的版、已记的 issue、已清的密钥……一句话带过即可。
- **⚠️ 需你决策**(真正开放、需要人拍板):未合的 PR、发了没部署的版、open 着等裁决的讨论 issue、push 了没开 PR 的分支……
- **💡 可选补充**(我没做、怕越界,交你判断):相关 tracker issue 要不要补记、memory 要不要存、scratchpad 临时文件要不要删……

结尾一句问用户:"哪些要我接着做?" —— 不预设动作、不批量执行。

## 已知约定 / 坑

- **只读**是硬规则。这个 skill 的价值在"如实盘点交人判断",一旦擅自 commit/合并/清理,既可能毁掉别人的并发工作,也越过了用户的判断权。
- **区分"我承诺的"与"可选的"**:本会话明确要做而没做完的 = 真遗留(⚠️);顺手能做但没人要求的(补 tracker、写 memory)= 可选(💡)。别把可选项渲染成必须项。
- **密钥/敏感产物优先查**:临时私钥、含明文口令的文件若残留是安全问题,务必单独点出。
- **发版≠部署**:tag/release 了不等于部署到目标机。若本会话既发版又计划部署,部署没做就是遗留。
- 公司仓 gh 无 token 会吞数据或报权限错,先按上文设 `GH_TOKEN`。
